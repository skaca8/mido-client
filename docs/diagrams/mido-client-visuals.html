화면 2. 인터셉터 체인 — 요청/응답 흐름

로깅이 평문 body 를 보고, 네트워크에는 압축본이 흐르는 의도된 순서

sequenceDiagram autonumber participant SVC as Service participant CUSTOM as 커스텀 인터셉터 participant LOG as MidoLoggingInterceptor participant GREQ as MidoGzipRequestInterceptor participant GRES as MidoGzipResponseInterceptor participant NET as 외부 서버 SVC->>CUSTOM: 요청 (평문 body) CUSTOM->>LOG: X-Request-Id 등 추가 LOG->>GREQ: 평문 body 그대로 로깅 GREQ->>GRES: body.length >= minSize 면 gzip 압축 GRES->>NET: Accept-Encoding gzip 강제 + 전송 NET-->>GRES: 응답 (Content-Encoding gzip 가능) GRES-->>LOG: 평문으로 디컴프레션 (cap 검사) LOG-->>GREQ: 평문 응답 로깅 GREQ-->>CUSTOM: 통과 CUSTOM-->>SVC: 비즈니스 코드로 반환 Note over GRES: maxDecompressedSize 초과 시\nIOException 즉시 발생 Note over LOG: log 레벨 off 면 양방향 모두 skip

등록 순서가 곧 호출 순서입니다. MidoClientFactory.createInterceptors() 의 주석 — "gzip 은 logging 뒤에 등록해야 로깅이 평문 body 를 본다" — 가 이 다이어그램의 핵심.
